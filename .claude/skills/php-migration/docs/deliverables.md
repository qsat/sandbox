# PHP移植プロジェクト 成果物リスト

AI agentが「人手レビューなしで移植 → 動作同等性が機械的に検証可能」な状態を終端条件とする。

---

## 1. 現状把握資料（Source-of-Truth）

| 成果物 | 内容 |
|--------|------|
| URLルーティング一覧 | ZF1のmodule/controller/action構成、URLパターン、HTTPメソッド、認可要件 |
| 画面一覧・画面遷移図 | 画面ID、対応URL、遷移トリガー、外部システム遷移点（問い合わせ・資料請求） |
| テンプレートインベントリ | .phtml/Smartyの親子関係、partial/layout継承、include依存、共通パーツ |
| 静的アセット一覧 | JS/CSS/画像、ビルド有無、SP特有のviewport/meta設定 |
| API利用カタログ | エンドポイント、リクエスト/レスポンス例、呼び出し画面、エラー応答 |
| セッション・Cookie・リクエストスコープ利用一覧 | 何をどこに保存しているか |
| ZF1 Helper/Plugin/Filter利用一覧 | 横断的関心事の棚卸し |

## 3. 変換マッピング資料（agentの「辞書」）

| 成果物 | 内容 |
|--------|------|
| ZF1 → Spring Boot コンポーネント対応表 | `Zend_Controller_Action`→`@Controller`、`Zend_View_Helper`→Thymeleaf Dialect、`Zend_Session_Namespace`→Spring Session、`Zend_Cache`→Spring Cache 等 |
| テンプレートエンジン変換ルール | Smarty/PHP構文 → Thymeleaf構文（`{if}`→`th:if`、`{foreach}`→`th:each`、partial include→`th:insert`/`th:replace`） |
| PHPイディオム → Javaイディオム変換ルール | null安全、配列→Collection、型強制、文字列処理 |
| APIクライアント生成規約 | OpenAPI仕様 or 手書き、DTO命名、エラー型統一、リトライ・タイムアウト方針 |
| 外部遷移点の取り扱い規約 | 問い合わせ・資料請求への遷移（URL組み立て、パラメータ受け渡し、戻り導線） |

---

## 4. 検証基準（agentが「完了」を判定するため）

| 成果物 | 内容 |
|--------|------|
| 画面単位の受け入れ基準（DoD） | 表示項目、状態、API呼び出し、遷移 |
| ゴールデンレスポンス／HTMLスナップショット | 同一入力でZF1版とSpring Boot版のレンダリング結果比較 |
| ビジュアル回帰テスト計画 | SP端末幅での比較 |
| APIモック・固定フィクスチャ | テストの再現性確保 |
| E2Eシナリオ集 | 主要導線：検索→詳細→問い合わせ遷移 |
| 非機能基準 | レスポンスタイム、同時接続、エラー率の閾値 |

## 5. agent作業プロセス定義

| 成果物 | 内容 |
|--------|------|
| 移植単位の定義 | 画面単位推奨（テンプレート＋Controller＋必要なService/Clientを1チケット化） |

---

## 着手優先順序

1. **1（インベントリ）** — 現状把握なしでは後続が机上論になる
2. **3（変換マッピング）** — agentの判断ルールを先に固める
3. **2（ターゲット設計）** — 1と3を踏まえて精緻化
4. **4（検証基準）** — 移植ループに入る前に合否基準を確定
5. **5（プロセス）** — 全体が見えた段階でプロセスを固定

最初の具体作業候補：**URLルーティング一覧** と **ZF1→Spring Boot対応表** のフォーマット策定。
